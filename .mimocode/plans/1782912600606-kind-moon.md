# QuikAssistant — Рекомендации по улучшению архитектуры

## Краткое описание

Анализ 24 Lua-файлов, тестов и веб-приложения. Рекомендации ранжированы по приоритету для live-системы, торгующей реальными деньгами. Каждое изменение должно быть обратимым и деплоимым инкрементально.

---

## ПРИОРИТЕТ 1 — Критические баги (деплоить немедленно)

### 1.1 Дублирование обработки сделок

**Проблема:** `StartEngine.lua:95-96` вызывает `TradeSave()` и `TradeClosePosition()` напрямую в main loop. Одновременно `Assistant.lua:122-141` определяет `N_OnNewTrade()` с идентичными вызовами. Но `N_OnNewTrade()` **нигде не вызывается** — это мёртвый код.

**Фактический путь выполнения:** QUIK callback `OnTrade()` → `N_Trades[]` → main loop итерирует `N_Trades` → вызывает `TradeSave`/`TradeClosePosition` напрямую (строки 95-96).

**Исправление:** Удалить `N_OnNewTrade()` из `Assistant.lua` (строки 122-141).

**Файлы:** `Assistant.lua`
**Риск:** НИЗКИЙ — удаление мёртвого кода, поведение не меняется
**Тест:** Запустить полный набор тестов; следить за корректностью записей в `MyTrades.csv`

### 1.2 Рекурсивный guard скрывает реальную проблему

**Проблема:** `Assistant.lua:144-206` — ручной счётчик рекурсии в `N_SetLimitOrder` (макс. 3). Цепочка: `N_SetLimitOrder → sendTransaction → OnTransReply → N_OnTransSendError → SetLimitOrdersWithError → SubmitOrders → N_SetLimitOrder`. На строке `TransactionHandler.lua:109` вызывается `SubmitOrders(orders)` внутри обработчика ошибок — это **осознанная повторная отправка** с скорректированной ценой.

**Исправление:** Заменить счётчик рекурсии на флаг `resubmit` в `N_SetLimitOrder`. По умолчанию `true`; при `false` обработчик ошибок не будет запускать повторную отправку. Сделает поведение явным.

**Файлы:** `Assistant.lua`, `TransactionHandler.lua`
**Риск:** СРЕДНИЙ — меняет поток обработки ошибок
**Тест:** Существующий тест `CheckOrder - UseFileParams + price below PRICEMIN`; добавить тест для сценария отклонение → повторная отправка

## ПРИОРИТЕТ 2 — Структурный долг (следующий спринт)

### 2.1 God Module: SubmittingOrders.lua (569 строк)

**Проблема:** 5 ответственностей в одном файле:

| Ответственность | Строки | Функции |
|---|---|---|
| Планирование сессий | 17-113 | `Initialization`, `SubmittingOrders` |
| Ожидание рыночных данных | 116-154 | `WaitForMarketData` |
| Оркестрация ордеров | 156-316 | `SubmittingOrdersRun` |
| CSV → загрузка ордеров | 319-416 | `LoadOrdersFromFile` |
| Закрытие позиции по сделке | 503-569 | `TradeClosePosition` |

**Исправление:** Разделить на 3 модуля:
1. **`SessionScheduler.lua`** — состояние сессий (флаги, таймеры), чистая конечный автомат без QUIK API
2. **`OrderLoader.lua`** — `LoadOrdersFromFile()` с парсингом CSV, валидацией строк
3. **`SubmittingOrders.lua` (урезанный)** — оркестратор `SubmittingOrdersRun()`, координирует загрузку и отправку (~200 строк)

**Файлы:** Новый `SessionScheduler.lua`, новый `OrderLoader.lua`, изменённый `SubmittingOrders.lua`
**Риск:** НИЗКИЙ — чистый рефакторинг, поведение одинаковое
**Тест:** Интеграционные тесты для каждого брокера

### 2.2 Загрязнение глобального пространства

**Проблема:** `_initSettingGlobals()` в `Setting.lua` копирует 15+ значений `Config.*` в голые глобалы. `Constants.lua` продвигает 8 констант в глобалы. ~40+ голых глобальных переменных.

**Исправление (3 фазы):**

**Фаза A (не ломает):** Заменить все чтения голых глобалов на `Config.*`:
- `TradeSave.lua` читает `Broker` → `Config.Broker`
- `SubmittingOrders.lua` читает `AccountCode` → `Config.AccountCode`, `ClientCode` → `Config.ClientCode`
- `log.lua` читает `Broker` → `Config.Broker`

**Фаза B (после A):** Удалить `_initSettingGlobals()` целиком.

**Фаза C:** Вынести lifecycle-флаги (`isRun`, `transId`, `N_TransReplies`, `N_Orders`, `N_Trades`) из `StartEngine.lua` в `EngineState.lua`.

**Файлы:** Все файлы, читающие голые глобалы + `Setting.lua`
**РISK:** СРЕДНИЙ — механические замены, каждая переменная мигрируется отдельно
**Тест:** Полный набор тестов после каждой миграции переменной

### 2.3 Несоответствие возвратов модулей

**Проблема:**
- `TableOrders.lua:145` возвращает `TableOrders` — неопределённый идентификатор (возвращает `{}`)
- `TableConstructor.lua:224` возвращает `TableConstructor` — неопределённый (возвращает `{}`). Полезный экспорт — `QTable`.

Модули работают только потому, что функции регистрируются как глобалы при `require`.

**Исправление:**
- `TableOrders.lua`: Экспортировать таблицу `{ RefreshTableOrdersControl, ClearTableOrdersControl }`
- `TableConstructor.lua`: Возвращать `QTable`

**Файлы:** `TableOrders.lua`, `TableConstructor.lua`, все вызывающие
**Риск:** СРЕДНИЙ — существующий код продолжает работать через глобалы
**Тест:** Проверить создание/отображение таблиц в QUIK

**Файлы:** Новый `FormatUtils.lua`, `TableConstructor.lua`, `LuaIndicators/PositionInfo.lua`, `Setting.lua`
**Риск:** НИЗКИЙ

### 3.2 Пробел интеграции веб-приложения

**Проблема:** Flask читает CSV-файлы, написанные Lua-системой. Нет общей схемы. Если Lua изменит порядок колонок — веб-приложение молча сломается.

**Исправление:** Добавить `CSV_SCHEMA.md` с точным форматом каждого CSV. Рассмотреть `schema.json` для валидации с обеих сторон.

**Файлы:** Новый `CSV_SCHEMA.md`, `web/csv_handler.py`
**Риск:** НИЗКИЙ

### 3.3 Dependency Injection для BrokerAdapter

**Проблема:** Все сервисные модули делают `require("BrokerAdapter")` на уровне модуля. Тесты работают через monkey-patching глобальных функций QUIK — хрупко.

**Исправление:** Передавать `BrokerAdapter` как параметр в конструкторы/функции модулей.

**Файлы:** Все сервисные модули + тестовый харнесс
**Риск:** ВЫСОКИЙ — меняет паттерн инициализации модулей; делать только после стабилизации Приоритетов 1-2

## Последовательность миграции

```
Неделя 1:  Деплой 1.1 (удаление мёртвого кода) + 3.1 (очистка неиспользуемых глобалов)
           ↓ Верификация на продакшене (2 сессии)
Неделя 2:  Деплой 2.2 Фаза A (миграция Config.*, по переменной)
           ↓ Тест-сьют после каждой переменной
Неделя 3:  Деплой 2.1 (разделение SubmittingOrders) + 2.3 (возвраты модулей)
           ↓ Интеграционные тесты для каждого брокера
Неделя 4:  Деплой 1.2 (исправление guard рекурсии) + 2.2 Фаза B (удаление _initSettingGlobals)
           ↓ Полная регрессия
Неделя 5+: 3.2 (схема CSV), 3.3 (DI) по мере возможности
```

Каждый деплой — отдельный коммит с чёткой точкой отката. Система не должна быть дальше одного изменения от известно-рабочего состояния.

## Ключевые находки

- `N_OnNewTrade()` в `Assistant.lua` — мёртвый код, нигде не вызывается
- Рекурсивный guard в `N_SetLimitOrder` — не баг, а маска осознанной логики повторной отправки
- `_initSettingGlobals()` создаёт 15 избыточных копий `Config.*`, доступных через `require("Config")`
- `TableOrders.lua` возвращает неопределённый идентификатор (пустая таблица)
- `comma_value`, `round`, `format_num` дублируются идентично в `TableConstructor.lua` и `PositionInfo.lua`
- `log.lua` читает голый глобал `Broker` — сломается при удалении глобалов без обновления log.lua
